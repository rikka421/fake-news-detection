# 前端接口对接说明

## 1. 启动后端服务

先训练并导出服务模型：

```bash
python scripts/train_serving_model.py
```

再启动 API：

```bash
python -m data_analysis_progress.api
```

默认地址：`http://localhost:8000`

## 2. 接口列表

### GET /health

- 用途：健康检查
- 返回示例：

```json
{
  "status": "ok"
}
```

### GET /model-info

- 用途：获取当前模型训练信息与指标
- 返回字段：
  - `metadata`: 数据集、切分方式、样本数等
  - `metrics`: validation/test 指标

### POST /predict

- 用途：单条新闻分类
- 请求体：

```json
{
  "title": "标题文本",
  "content": "正文文本",
  "meta_description": "可选",
  "summary": "可选"
}
```

- 返回：

```json
{
  "label": "reliable"
}
```

### POST /predict-batch

- 用途：批量分类（前端批量测试用）
- 请求体：

```json
{
  "texts": ["新闻文本1", "新闻文本2"]
}
```

- 返回：

```json
{
  "labels": ["fake", "reliable"]
}
```

## 3. 前端调用建议

- 当 `GET /health` 返回 `degraded` 时，提示后端模型未准备好。
- 对 `POST /predict` 的输入做最小长度校验，避免全空文本。
- 首次联调优先用 `/model-info` 确认后端模型版本和指标。

## 4. LLM 方案说明（实验）

项目中已补充 `Qwen/Qwen2.5-0.5B-Instruct` 零样本实验脚本：

```bash
python scripts/eval_llm_zero_shot.py --model Qwen/Qwen2.5-0.5B-Instruct --dataset balanced_large --method generation --max-samples 1
```

实测结果：

- CPU 环境：
  - `generation` 单条约 7.81 秒，10k 预计约 21.69 小时
  - `likelihood` 单条约 181 秒，10k 预计约 502.87 小时
- GPU 环境（RTX 3050 + cu126 + FP16）：
  - 修复长文本截断策略后（保证 `Label:` 指令未被截断），200 条随机样本平均约 0.588 秒/条
  - 吞吐约 1.70 条/秒，10k 预计约 98.08 分钟
  - 标签提取命中率约 87.5%
  - Accuracy 约 0.070，Macro F1 约 0.0469

因此前端实时服务仍建议优先接当前传统模型 API（准确率显著更高）。若要达成“10k 在 1 小时内”，并且必须使用 LLM，应优先：

- 使用当前 `/predict` 与 `/predict-batch`（SVM 推理服务）
- 或将 LLM 部署到 GPU 推理平台，并做批处理与并发优化
