# 小组项目技术报告（简版）

## 一、项目目标与分工

本小组项目目标是构建一个可用的假新闻分类系统，并对外提供网页可调用的推理接口。

本阶段已完成的核心交付：

- 数据采样与清洗流程；
- 多模型实验对比与可用分类模型选型；
- 模型接口封装（HTTP API）；
- 前端联调接口文档。

## 二、数据处理与实验流程

### 1. 数据来源

- 使用 FakeNewsCorpus 子集；
- 采用固定规模与均衡采样策略，确保实验可复现。

### 2. 数据策略

- 训练开发主要使用 `balanced_medium`（1000 条）进行快速迭代；
- 统一使用 `title_content` 文本拼接方式；
- 启用 `mask_label_tokens` 降低标签词泄漏风险；
- 使用 `grouped_title_content` 切分减少模板穿越。

### 3. 模型对比

已对 Logistic Regression、SVM、RandomForest、Naive Bayes、XGBoost、LightGBM、CNN、Transformer 做统一评测。

在当前实验条件下，传统机器学习模型整体优于当前轻量深度模型。

## 三、可用模型选型

### 1. 最终服务模型

当前接口默认采用 **SVM + TF-IDF** 方案，原因：

- 指标稳定且接近最优；
- 推理速度快，适合网页实时调用；
- 模型体积和部署成本可控；
- 训练流程清晰，便于后续替换。

### 2. 服务模型导出

通过 `scripts/train_serving_model.py` 训练并导出：

- 模型文件：`models/serving/fakenews_svm_pipeline.joblib`
- 同时保存训练元信息和验证/测试指标。

当前服务模型指标（balanced_medium, grouped_title_content 切分）：

- Validation Accuracy: 0.6554
- Validation Macro F1: 0.6466
- Test Accuracy: 0.6577
- Test Macro F1: 0.6662

## 四、接口封装设计

### 1. 技术选型

- 后端框架：FastAPI
- 数据交换：JSON
- 接口风格：REST

### 2. 核心接口

- `GET /health`：服务健康检查
- `GET /model-info`：模型信息与指标
- `POST /predict`：单条文本分类
- `POST /predict-batch`：批量文本分类

接口已做本地联调验证：

- `/health` 返回 `ok`；
- `/model-info` 可返回模型元数据与指标；
- `/predict` 与 `/predict-batch` 可返回分类标签数组或单标签结果。

### 3. 前端协作方式

- 前端通过 `/predict` 获取分类结果标签；
- 前端可在页面加载时调用 `/health` 与 `/model-info` 做状态展示。

## 五、关键实现过程

1. 完成多模型训练框架并输出结构化评测报告；
2. 扩展常用 ML 模型与深度模型对比；
3. 针对深度模型尝试预训练 tokenizer/embedding 初始化并进行效果验证；
4. 选定可上线模型并封装 API；
5. 输出前端对接说明和项目报告。

## 六、当前结果与结论

- 项目已具备“可训练、可评估、可接口调用”的完整闭环；
- 前端同学可基于现有 API 直接开发网页；
- 已完成 `balanced_large`（10000 条）全模型统一评测。

### 1. balanced_large 全模型结果（test）

| 模型 | Accuracy | Macro F1 |
|------|----------|----------|
| XGBoost | 0.8072 | 0.8137 |
| LightGBM | 0.8059 | 0.8108 |
| RandomForest | 0.7970 | 0.8015 |
| SVM | 0.7546 | 0.7566 |
| Logistic Regression | 0.7396 | 0.7448 |
| CNN | 0.6972 | 0.7181 |
| Transformer | 0.6500 | 0.6619 |
| Naive Bayes | 0.6446 | 0.6513 |

结论：在当前设定下，树模型（XGBoost/LightGBM/RandomForest）整体表现最佳。

### 2. <=0.5B LLM 实验结果

模型：`Qwen/Qwen2.5-0.5B-Instruct`

- CPU：`generation` 约 7.81 秒/条，10k 约 21.69 小时；
- 诊断发现：若长文本被直接截断，提示尾部 `Label:` 可能被切掉，模型会偏向“续写正文”而不是“输出标签”；
- 修复截断策略后（保留标签指令），GPU（RTX 3050 + FP16）在 200 条随机样本上约 0.588 秒/条，10k 预计 98.08 分钟；
- 标签提取命中率约 87.5%，但零样本分类效果仍较弱：Accuracy 0.070，Macro F1 0.0469。

结论：即使修复评测流程，当前 LLM 仍在速度与准确率上落后于主模型方案，不适合作为当前线上主模型。
